# 付録 B: AI セキュリティ管理策の目録

## 目的

この付録は、AISVS 要件全体で義務付けられているセキュリティ管理策を、開発者向けに統合した一覧表です。管理策は管理策ファミリーごとにグループ化されているため、実装者はどの章で定義されているかに関わらず、関連するすべての防御策を 1 か所で見つけることができます。また、各管理策は、それを義務付けている AISVS 要件 ID にリンクされています。

この一覧表は、規範的なものではありません。既存の要件を実装の容易性を考慮して再編成したものであり、要件の追加、削除、変更は一切行っていません。要件の章（C1～C12）が、引き続き信頼できる情報源となります。要件 ID は、標準的な `C{章}.{節}.{要件}` の形式で記述されています（例: `C5.1.1`）。標準内の番号付き要件はすべて、必ず 1 つの管理策ファミリーに該当するため、一覧表の網羅性を各章と照らし合わせて確認できます。

---

## AD.1 認証とアイデンティティ

アクセスを許可する前に、ユーザー、エージェント、サービス、エッジ デバイス、および MCP クライアント/サーバーのアイデンティティを検証します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| 高リスク AI 操作（モデルのデプロイメント、重みのエクスポート、学習データへのアクセス、本番構成の変更）に対するステップ アップ認証を実施します。 | C5.1.1 |
| フェデレーションまたはマルチ システム エージェント認証のための、有効期限が短く、スコープが最小限の暗号署名付きトークンを使用します。 | C5.1.2 |
| エッジ AI デバイスから中央インフラストラクチャへの強力な認証を実施します。 | C4.3.1 |
| エージェント インスタンスごとに一意の暗号アイデンティティを使用し、下流のシステムに対して第 1 級のプリンシパルとして認証します。 | C9.4.1 |
| エージェントのアイデンティティ資格情報を定期的にローテーションします。 | C9.4.3 |
| MCP リクエストごとのアクセス トークンを検証します（トランスポート セキュリティのみではない）。 | C10.2.1 |
| OAuth 2.1 に基づく MCP アクセス トークンのクレーム (発行者、対象者、有効期限、スコープ）を検証します。 | C10.2.2 |
| MCP リソース サーバーはアクセス トークンまたはユーザー資格情報を保存または永続化しません。 | C10.2.3 |
| セッション終了時にすべての MCP セッション アーティファクトを削除します。 | C10.2.6 |
| クライアント アクセス トークンを下流の API に渡しません。 | C10.2.7 |
| 送信者制約付き MCP アクセス トークン (mTLS または DPoP)を使用します。 | C10.3.5 |

**よくある落とし穴**: エージェント間呼び出しにエンドユーザーの資格情報を再利用しています。侵害が疑われる場合にエージェントの資格情報をローテーションしていません。リクエストごとのトークン確認 (validation) の代わりにトランスポート セキュリティを扱っています。

## AD.2 認可とアクセス制御

モデルが上書きできないポリシーを使用して、ユーザー、エージェント、ツール、リソース全体にわたるアクセス決定を強制します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| すべての AI リソース（データセット、エンドポイント、ベクトル コレクション、エンベディング インデックス、計算）に対する明示的な許可リストとデフォルト拒否によるアクセス制御を実施します。 | C5.2.1 |
| サービス アカウントだけでなく、各取得およびアセンブリ段階でエンドユーザーの認証コンテキストを強制します。 | C5.2.2 |
| 推論後のフィルタリングにより、要求者が受信する権限のないデータは応答から除外します。 | C5.2.4 |
| エージェント実行環境から分離されたポリシー決定ポイントを設けます。 | C5.2.5 |
| 自動有効期限付きの、モデル重み、学習パイプライン、および本番構成へのジャスト イン タイム特権アクセスを実施します。 | C5.2.6 |
| エージェント アクションのきめ細かな、ランタイム強制の認可（どのツール、どのパラメータ値を使用するか）を実施します。| C9.5.1 |
| 完全性が保護されスコープが限定された委任トークンがすべての下流の呼び出しに伝播します。 | C9.5.2 |
| アクセス制御の決定は、アプリケーション ロジックまたはポリシー エンジンによって強制され、モデルによって強制されません。 | C9.5.3 |
| 明示的な認可ポリシーによって制限されるエージェント間タスク委任を実施します。 | C9.5.5 |
| 長時間実行セッションにおけるすべての特権アクションに対するバックエンド認可の再評価します。 | C9.5.6 |
| スコープでフィルタリングされた MCP ツールを検出します（tools/list は承認されたツールのみを返します）。| C10.2.4 |
| ツールと指定された引数値の両方を確認 (validate) する、呼び出しごとの MCP アクセス制御を実施します。 | C10.2.5 |

**よくある落とし穴**: 呼び出し元の権限ではなく、サービス アカウントの権限に依存しています。モデルが生成した出力に基づいて認証を行っています。セッション中にコンテキストが変更された場合でも、認証を再確認していません。

## AD.3 データの分類およびテナントの分離

AI 固有の変換処理や共有インフラストラクチャを通過する際、データは認証境界とテナント境界内に保持されます。

| 管理策/手法 | 要件 ID |
| --- | --- |
| 機密データは、モデルの重みに永続化するのではなく、取得パイプラインを通じて提供されます。 | C5.2.3 |
| 分類ラベルは、下流のリソース（エンベディング、プロンプト キャッシュ、モデル出力）に伝播されます。 | C5.2.7 |
| 共有モデル サービング（微調整（ファイン チューニング）、推論、エンベディング操作）におけるテナント間分離を実施します。 | C5.3.1 |
| 共有コンピューティング（ハードウェア パーティショニング、機密コンピューティング、専用割り当て）におけるテナント間分離を実施します。 | C5.3.2 |

**よくある落とし穴**: データがエンベディング化されたりキャッシュされたりする際に分類ラベルが削除されています。共有推論キャッシュにおけるサイドチャネル攻撃に対して、論理的なマルチ テナンシーで十分だと想定しています。

## AD.4 暗号化およびデータ保護

保存時、転送時、およびモデルの観測可能なコンテキストにおけるデータと機密情報を保護します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| 学習データの保存および転送時の完全性保護を実施します。 | C1.1.3 |
| ラベル付けアーティファクトで使用する前に、ラベル内の機密情報を秘匿化、匿名化、または暗号化します。 | C1.2.3 |
| ハードウェア ベースの鍵ストアまたはセキュア エンクレーブを使用して、ローカルに保存されたモデルの重みと機密パラメータを暗号化します。 | C4.3.4 |
| モバイル、IoT、または組み込みアプリにパッケージ化されたモデルを保存時に暗号化します。復号は信頼できるランタイムまたはセキュア エンクレーブ内でのみ行います。 | C4.3.5 |
| モデルの観測可能なコンテキスト（コンテキスト ウィンドウ、システム プロンプト、ツール呼び出しパラメータ）から機密情報と資格情報を保護します。 | C9.5.4 |

**よくある落とし穴**: データベースは暗号化しますが、モデルのチェックポイントやエンベディングは暗号化していません。アプリ パッケージからモデルの重みを抽出可能です。ツール呼び出しパラメータ内に API キーを公開しています。

---

## AD.5 完全性、署名および来歴

モデル、アーティファクト、メッセージ、ツール定義、および生成されたメディアの真正性を検証し、改ざんを検出します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| 学習データの不正な変更または破損に対する完全性監視を実施します。 | C1.1.4 |
| アーティファクトのラベル付けにおける暗号学的完全性を確認します。 | C1.2.2 |
| すべてのモデル アーティファクト（重み、構成、トークナイザー、ベース モデル、微調整（ファイン チューン）、アダプタ、安全性/ポリシーモデル）の暗号署名を実施します。 | C3.1.2 |
| デプロイメントの承認時およびロード時の署名検証を実施します。 | C3.1.3 |
| ロード前にデバイス上で署名またはチェックサム検証を行う署名付きエッジ/モバイル モデル パッケージを実装します。 | C4.3.2 |
| 否認防止のため、エージェントが開始したアクションを各実行チェーンステップに暗号学的にバインドします。 | C9.4.2 |
| 呼び出し間で保持されるエージェント状態の完全性保護を実施します。 | C9.4.4 |
| リプレイ攻撃対策として、固有のノンスとタイムスタンプを付与した署名付き MCP ツール応答を実装します。 | C10.4.6 |
| ツール定義のスナップショット作成と、呼び出し前に変更があった場合の再承認の義務付けます。 | C10.4.8 |
| AI 生成メディアに透かしを付与し、AI 生成であることを証明します。 | C7.4.4 |

**よくある落とし穴**: 不変のダイジェストではなく可変タグを使用しています。MCP 呼び出し間でツール定義を再検証していません。ツール応答にリプレイ攻撃対策がありません。

## AD.6 入力の確認 (validation) と無害化

ツール、MCP、取得コンテンツを含むすべての入力を、モデルまたは下流のシステムに到達する前に確認 (validate)、正規化、および制約します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| トークン化またはエンベディング化の前に入力の正規化を適用します。 | C2.1.1 |
| エンコーディングおよび表現スマグリングの検出と軽減（正規化、厳密なスキーマ検証、ポリシー ベースの拒否、または明示的なマーキング）を実施します。 | C2.1.2 |
| プロンプト インジェクション検出のルール セットまたは分類器による信頼できない入力のスクリーニングとブロックを実施します。 | C2.1.3 |
| コンテキスト ウィンドウを超えるコンテンツを拒否する（切り捨てない）入力長制御を実施します。 | C2.1.4 |
| すべての入力に対する許可リスト文字セット制限を実施します。 | C2.1.5 |
| 命令階層の強制（システムおよび開発者メッセージがユーザーおよび信頼できない入力を上書きする）を実施します。 | C2.1.6 |
| リテラル文字としてエンコードされ、コンテキストに挿入できない予約済み特殊トークンを使用します。 | C2.1.7 |
| 多段式脱獄（ジェイルブレイク）パターンを検出します。 | C2.1.8 |
| 非テキスト入力（画像、動画、音声）に対する敵対的摂動、ステガノグラフィ、および隠しコンテンツをチェックします。 | C2.2.3 |
| クロス モーダル協調攻撃を検出します。 | C2.2.4 |
| ツール出力のスキーマ検証を実施します。 | C9.3.2 |
| インストールまたは呼び出し前に、モデル出力で指定された外部リソースを承認済み許可リストまたはレジストリと照合して検証します。 | C9.3.7 |
| モデル コンテキストへの挿入前に、MCP 応答スキーマを確認 (validate) します。 | C10.4.1 |
| モデル コンテキストへの挿入前に、MCP 応答の間接プロンプト インジェクションのスクリーニングを実施します。 | C10.4.2 |
| 認識されない、またはサイズが大きすぎる MCP 関数呼び出しパラメータを拒否します。 | C10.4.3 |
| 厳密な MCP スキーマ検証を実施します。 | C10.4.4 |
| MCP ペイロードの最大サイズを制限します。 | C10.4.5 |
| 推論前の外部入力または信頼できない入力に対する異常を検出します。 | C11.4.1 |
| 異常と判定された入力に対するアクションのゲート化を実施します。 | C11.4.2 |

**よくある落とし穴**: 画像/音声チャネルを無視してテキスト モダリティのみを検証しています。セマンティック検出を行わずに正規表現のみに依存しています。ツールおよび MCP の出力がエージェント コンテキストに再入力される前に検証を行っていません。

## AD.7 インバウンド コンテンツおよびポリシーのスクリーニング

プロンプトと学習コンテンツは、モデルまたは学習パイプラインに到達する前に、ポリシーに基づいてスクリーニングされます。

| 管理策/手法 | 要件 ID |
| --- | --- |
| 設定可能な閾値に基づいて受信コンテンツを分類し（暴力、自傷行為、ヘイト スピーチ、性的表現など）、モデル コンテキストの前に拒否または無害化します。 | C2.2.1 |
| サポートされていない言語のコンテンツ分類を評価します。 | C2.2.2 |
| 学習前に、許可されていないコンテンツを検出して削除します。 | C1.3.4 |

**よくある落とし穴**: 1つの言語のみに調整された分類器をデプロイしています。プロンプトはスクリーニングしますが、学習コーパスはスクリーニングしていません。

## AD.8 出力処理および安全性

モデルの出力は、ユーザーまたは下流のシステムに到達する前に、制約、フィルタリング、および検証されます。

| 管理策/手法 | 要件 ID |
| --- | --- |
| モデル出力のスキーマを検証します（不一致の場合は拒否します）。| C7.1.1 |
| 生成出力の長さ制限と終了制御を実施します。| C7.1.2 |
| 生成された回答の信頼度または不確実性を推定します。| C7.2.1|
| 信頼度が既定のしきい値を下回った場合に自動ブロックまたはフォールバックを実施します。| C7.2.2 |
| ポリシーによって高リスクと分類された応答に対する追加検証ステップを設けます。| C7.2.3 |
| 応答をスキャンし、既定の有害コンテンツ カテゴリをブロックする自動分類器を設けます。 | C7.3.1 |
| システム プロンプトの内容またはバックエンド データを開示する応答を検出しブロックします。| C7.3.2 |
| モデル生成出力によるアウトバウンド要求のトリガーを防止します。| C7.3.3 |
| 隠蔽、エンコード、または誤解を招く出力（同形異義語、書式設定、メタデータ、構造化フィールド）を検出します。| C7.3.4 |

**よくある落とし穴**: バッチ モードでは停止シーケンスを適用しますが、ストリーミング出力では適用していません。システム プロンプトを言い換えて漏洩させています。プロバイダーが信頼度スコアを公開していないにもかかわらず、信頼度スコアが利用可能であるかのように扱っています。

## AD.9 レート制限、予算およびリソースの制御

不正利用、暴走実行、サービス拒否攻撃、モデル抽出を防止するために、消費量を制限します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| ツールごとのクォータとタイムアウト（CPU、メモリ、ディスク、出力、実行時間）を設けます。| C9.1.1 |
| ランタイムによって強制される実行ごとの予算（最大再帰深度、トークン使用量、金銭的支出）を設けます。| C9.1.2 |
| 抽出脅威モデルに合わせて設定された、プリンシパルごとおよびグローバルな推論レート制限（一般的な API スロットリングではありません）を設けます。| C11.2.2 |

**よくある落とし穴** エンドポイントごとにレート制限を設定していますが、エージェント セッションごとに設定していません。予算設定時にツールのファン アウト（訳注: 大きなタスクを小さなサブ タスクに分割すること）を考慮していません。抽出防御を通常のスロットリングとして扱っています。

---

## AD.10 サンドボックス化とワークロードの分離

モデル、ツール、エージェント、およびハードウェア ワークロードを分離することで、障害を封じ込め、横方向への移動（ラテラル ムーブメント）を防ぎます。

| 管理策/手法 | 要件 ID |
| --- | --- |
| 分離されたサンドボックスで AI モデルを実行します。 | C4.1.1 |
| デシリアライゼーション中にコード実行を許可しないシリアライゼーション形式の許可リストを設けます。 | C4.1.2 |
| モデルのロード前にワークロードを認証（アテステーション）します。 | C4.1.3 |
| 分離された実行により、ラインタイムにおけるモデル重みを保護する機密性推論を実施します。 | C4.1.4 |
| ハードウェアによる分離、メモリ暗号化、および完全性保護を備えた信頼できる実行環境を実装します。 | C4.2.2 |
| 各ワークロード前にハードウェア認証（アテステーション）による GPU の完全性を確認 (validate) します。 | C4.2.3 |
| ジョブ間の無害化を伴う GPU メモリのパーティショニングを実施します。 | C4.2.4 |
| バージョン固定、署名、ブート認証済みのアクセラレータ ファームウェアを使用します。 | C4.2.1 |
| エッジ推論ランタイムにおけるプロセス、メモリ、ファイル アクセスを分離します。 | C4.3.3 |
| 各ツールまたはプラグインに対して最小権限サンドボックス化または分離を実施します。 | C9.3.1 |
| 必要な権限、リソース制限、出力検証要件を宣言するツールマニフェスト | C9.3.3 |
| 宣言されたツール マニフェストの権限と制限をランタイムで適用します。 | C9.3.4 |
| 信頼できないデータ処理とツール呼び出し機能を分離します。 | C9.3.5 |
| 信頼できないツール出力処理とエージェント操作をアーキテクチャ上で分離します。 | C9.3.6 |
| ローカルで起動された MCP サーバーに対して最小権限サンドボックス化します（ファイル システム、ネットワーク、システム アクセスの制限）。 | C10.1.3 |
| 環境境界（開発、ステージング、本番）を越えて共有されない AI 固有のランタイム コンポーネントを実装します。 | C3.4.1 |
| 本番環境から隔離された学習および微調整（ファイン チューニング）環境を設けます。 | C3.4.2 |

**よくある落とし穴**: 開発環境と本番環境間でインフラストラクチャを共有しています。ツール サンドボックスに必要以上の権限を与えています。信頼できないデータ処理がツール呼び出しパスに到達することを許容しています。

## AD.11 ネットワークおよび出口制御

AI ワークロードおよび MCP 統合におけるネットワーク境界、トランスポート セキュリティ、およびトラフィック フローを制御します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| リモート MCP トランスポートのための認証付き暗号化ストリーミング HTTP を使用します。 | C10.3.1 |
| stdio MCPトランスポートを制御されたローカル環境に限定します。 | C10.3.2 |
| HTTP ベースのトランスポートにおける独立した Origin および Host ヘッダー検証を実施します（DNS リバインディング防御）。 | C10.3.3 |
| MCP クライアントの最小プロトコル バージョンを強制します（ダウングレード防御）。 | C10.3.4 |
| アクセラレータ相互接続を承認済みトポロジおよび認証済みエンドポイントに限定します。 | C4.2.5 |

**よくある落とし穴**: stdio または SSE トランスポートをローカルホスト外に公開しています。Origin/Host 検証をスキップして DNS リバインディングを有効にしています。ダウングレードされたプロトコル バージョンを受け入れています。

## AD.12 サプライチェーンおよびアーティファクトの完全性

モデル、データセット、フレームワーク、および MCP コンポーネントの出所と真正性を検証し、AI 部品表（BOM）を維持します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| デプロイされたすべてのモデル アーティファクトとその出所を記載したモデル レジストリのインベントリを実装します。 | C3.1.1 |
| インポート前に、モデルの悪意のあるコードに対するスキャンを実施します。 | C6.1.1 |
| 承認されたソースからのみモデルの重み、データセット、および微調整（ファイン チューニング）アダプタをダウンロードします。 | C6.1.2 |
| すべてのサードパーティ製モデル アーティファクトの完全性を検証します。 | C6.1.3 |
| 開発段階から次の段階に進める前に、動作受け入れテスト スイートに合格することを必要とします。 | C6.1.4 |
| モデル アーティファクトごとのバージョン管理された機械可読な AI BOM（データセット、重み、ライセンス、データ出所ステートメント）を実装します。 | C6.2.1 |
| デプロイ前に AI BOM の暗号署名を実施します。 | C6.2.2 |
| コンポーネント メタデータが欠落している場合に、AI BOM 完全性チェックによってビルドを失敗させます。 | C6.2.3 |
| MCP コンポーネントは信頼できるソースからのみ取得し、暗号的に検証します。 | C10.1.1 |
| 許可リストに登録された MCP サーバーのみを使用します。 | C10.1.2 |

**よくある落とし穴**: AI BOM を署名付きバージョン管理されたアーティファクトではなく静的な文書として扱っています。バックドアがないか事前学習済み重みをスキャンしていません。承認されていないレジストリからモデルを取得しています。

## AD.13 モデルのライフサイクル、デプロイ、ロールバック

モデルの検証、デプロイ、ロールバック、および微調整（ファイン チューニング）パイプラインの完全性を管理します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| デプロイ前の自動入力検証、安全性評価、および出力無害化のテストを実施します。 | C3.2.1 |
| デプロイ前に、学習後の量子化処理を受けたモデルを、同じ安全性および整合性（アライメント）テストスイートに対して再評価します。 | C3.2.2 |
| プロバイダ モデル、バージョン、またはルーティングの変更によってトリガーされるセキュリティを再評価します。 | C3.2.3 |
| 自動ロールバック トリガーを備えたロールアウト機構を実装します。 | C3.3.1 |
| ロールバック時に完全なモデル状態を復元します。 | C3.3.2 |
| 並列実行中のモデル バージョンの分離されたランタイム状態を実装します。 | C3.3.3 |
| 学習実行前にバージョン管理され、完全性が検証された RLHF 報酬モデルを実装します。 | C3.5.1 |
| RLHF ステージにおける報酬ハッキングまたは報酬モデルの過剰最適化を検出します。 | C3.5.2 |
| 多段階の微調整（ファイン チューニング）パイプラインにおいて、ステージごとに完全性を検証します。 | C3.5.3 |
| 微調整（ファイン チューニング）チェックポイントを個別のアーティファクトとして登録します。 | C3.5.4 |

**よくある落とし穴**: ロールバックが必要になる前にテストを行っていません。廃止されたモデル アーティファクトをサービング キャッシュに残しています。報酬モデルをバージョン管理され検証済みのアーティファクトではなく、静的なインフラストラクチャとして扱っています。

## AD.14 学習データの完全性およびガバナンス

改ざん、汚染、破損を検出・追跡できるように、学習データのソース、検証、文書化を行います。

| 管理策/手法 | 要件 ID |
| --- | --- |
| 明示された目的に必要な特徴、属性、フィールドのみにデータを最小化します。 | C1.1.1 |
| すべての学習データソースの最新のインベントリ（出所、責任者、ライセンス、収集方法、使用制限、処理履歴）を実装します。 | C1.1.2 |
| 使用状況の特定と不正使用の検出のためにデータセットの透かしを実装します。 | C1.1.5 |
| アノテーションの作成、変更、承認を制限できるユーザーを制限したラベリング プラットフォームのアクセス制御を実施します。 | C1.2.1 |
| 学習および微調整（ファイン チューニング）パイプラインにおける汚染検出を実施します。 | C1.3.1 |
| 自動生成されたラベルに対する信頼度閾値と一貫性をチェックします。 | C1.3.2 |
| セキュリティ関連の意思決定に使用されるモデルのバイアス評価を実施します。 | C1.3.3 |
| クリーン ラベル汚染攻撃に対する防御策を実装します。 | C1.3.5 |
| データセットの系統（変換、拡張、マージ）を記録します。 | C12.5.1 |
| すべてのラベリング活動をログ記録します。 | C12.5.2 |
| 取り込まれたすべての文書へ書き込み時にタグ付け（ソース、作成者アイデンティティ、タイムスタンプ）します。 | C12.5.4 |

**よくある落とし穴**: 微調整（ファイン チューニング）データセットの汚染スキャンを行っていません。目的以上に多くの属性を収集しています。変換やマージの過程でデータセットの系統が失わています。

## AD.15 メモリ、エンベディング、および RAG のセキュリティ

ベクトル ストア、メモリ パイプライン、および検索拡張生成を、情報漏洩、汚染、および偽造された来歴から保護します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| テナントごとのベクトル識別子と名前空間の一意性を確保し、テナント間の衝突を防止します | C8.1.1 |
| 初回書き込み後の文書メタデータのタグの不変性を確保します。 | C8.1.2 |
| 検索操作にスコープ制約を適用します。 | C8.1.3 |
| エンベディング化前に機密フィールドを検出してマスキング、トークン化、または削除します。 | C8.2.1 |
| ベクトル化前に検索操作コンテンツを検出、拒否、または隔離します。 | C8.2.4 |
| 本番インデックスに入る前に外れ値ベクトルをフラグ付けして隔離します。 | C8.2.2 |
|エージェントまたはツールの出力が信頼できるメモリに書き込まれる前にソースを検証します。 | C8.2.3 |
| 新規メモリ書き込み時に矛盾をチェックし、競合が発生した場合はアラートをトリガーします。 | C8.2.5 |
| 有効期限切れのベクトルを検索結果から除外します。 | C8.3.1 |
| メモリ リセット機能を実装します。 | C8.3.2 |
| 隔離されたコンテンツをすべての検索結果から除外しながら保持します。 | C8.3.3 |
| RAG 応答をソース文書に紐付けます。 | C7.4.1 |
| RAGの属性は、モデルによって生成されたものではなく、検索メタデータから導出されます。| C7.4.2 |
| RAG クレームを、取得されたチャンクに追跡可能にします。 | C7.4.3 |

**よくある落とし穴**: 検証なしでツールの出力を信頼できるメモリに自動書き込みしています。有効期限切れまたは隔離されたベクトルを提供しています。検索結果のメタデータから引用を導き出すのではなく、モデルに引用を生成させています。

## AD.16 敵対的堅牢性およびプライバシー保護

回避、メンバーシップ推論、モデル反転、抽出、および改善ループの汚染に対するテストと防御を実施します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| 許可されていないコンテンツ カテゴリを抑制するために、整合性（アライメント）と安全性の学習または微調整（ファイン チューニング）を実施します。 | C11.1.1 |
| モデルの更新またはリリースごとに、バージョン管理された整合性（アライメント）テスト スイートを実行します。 | C11.1.2 |
| モダリティに関連する既知の敵対的攻撃手法に対する評価を実施します。 | C11.1.3 |
| 敵対的入力に対してモデルを堅牢化します。 | C11.1.4 |
| 有害コンテンツ率を測定し、閾値を超える回帰をフラグ付けする自動評価ツールを使用します。 | C11.1.5 |
| モデルから推論された機密属性を直接的に抑制します。 | C11.2.1 |
| 推論攻撃に悪用される可能性のある過信予測を低減するために、出力を較正します。 | C11.2.3 |
| 機密データセットでの学習のために差分プライバシーを最適化します。 | C11.2.4 |
| ランダム推測と同等の精度を示すメンバーシップ推論攻撃シミュレーションを実施します。 | C11.2.5 |
| 生のモデル出力はバックエンド以外には公開されず、外部から見える応答は抽出リスクに合わせて調整されます。 | C11.3.2 |
| 不正コピーを識別できるように、モデルに透かしまたはフィンガープリンティングを施します。 | C11.3.3 |
| 汚染検出と人間によるレビュー ゲートにより、安全性違反フィードバック パイプラインを保護します。 | C11.4.3 |

**よくある落とし穴**: 適応型攻撃を用いずに既知の脱獄（ジェイルブレイク）パターンのみをテストしています。モデル更新後に整合性（アライメント）スイートを再実行していません。生の信頼度ベクトルを公開し抽出を加速させています。

## AD.17 ログ記録および監査

フォレンジックによる再構築と説明責任の確保のため、セキュリティ関連イベントを十分なコンテキストと完全性をもって記録します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| セッション コンテキストと AI 固有のテレメトリを含む AI インタラクションをログ記録します。 | C12.1.1 |
| コンテンツ モデレーションの監査に十分な詳細度で、安全性フィルタリングとポリシー決定をログ記録します。 | C12.1.2 |
| 推論イベントのための構造化された相互運用可能なログ スキーマ（モデル識別子、トークン使用状況、プロバイダ、操作タイプ）を実装します。 | C12.1.3 |
| RAG パイプライン取得イベント（クエリ、取得ドキュメント、知識ソース）をログ記録します。 | C12.1.4 |
| セキュリティ上重要なプロアクティブなアクションの承認チェーンを記録する監査ログ（承認者のアイデンティティ、タイムスタンプ、パラメータ、結果）を生成します。 | C12.4.2 |
| キル スイッチの有効化とオーバーライド コマンドをログ記録します。 | C12.4.3 |
| すべてのモデル変更に関する不変の監査記録を実装します。 | C12.5.3 |

**よくある落とし穴**: プロンプトを秘匿化せずにログ記録しています。完全性保護のない可変ログ ストレージを使用しています。エージェントのアクションと承認は記録しますが、キル スイッチの起動など人間が開始したオーバーライドは記録していません。

## AD.18 監視、検出、インシデント対応

AI 固有の不正利用、逸脱、異常を検出し、インシデントに対応します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| ポリシー違反によってトリガーされる自動ツールで封じ込めます。 | C9.3.8 |
| クエリ パターン分析に基づく抽出試行検出器を実装します。 | C11.3.1 |
| 疑わしいモデル抽出の検出時にトリガーされる対応策を実装します。 | C11.3.4 |
| 脱獄（ジェイルブレイク）パターン、プロンプト インジェクション、敵対的入力に対して、シグネチャ ベースで検出しアラートを発行します。 | C12.2.1 |
| 振舞い異常（異常な会話パターン、過剰な再試行、体系的なプロービング）を検出します。 | C12.2.2 |
| AI 固有の脅威パターン（協調的な脱獄（ジェイルブレイク）試行、プロンプト インジェクション、システム プロンプトの抽出）に対するカスタム検出ルールを実装します。 | C12.2.3 |
| 問題のあるクエリのメタデータを含む抽出アラート イベントを実装します。  | C12.2.4 |
| 詳細なトークン使用状況（ユーザー、セッション、機能エンドポイント、チーム、ワークスペースごと）の特定します。 | C12.2.5 |
| 隠しチャネルおよびコマンド＆コントロール指標のために、LLM API トラフィックを監視します。 | C12.2.6 |
| 入力タイプに合わせた手法（表形式データの場合は KS 検定または PSI、テキスト/画像データの場合はエンベディング距離）を用いて、データ ドリフトを検出します。 | C12.3.1 |
| モデル出力の幻覚（ハルシネーション）を検出し監視します。 | C12.3.2 |
| 幻覚（ハルシネーション）発生率を時系列指標としての追跡します。 | C12.3.3 |
| 説明のつかない動作変化と緩やかな運用ドリフトを区別します。 | C12.3.4 |
| 自律アクションをトリガーするために、セキュリティ評価と脅威状況分析を実施します。 | C12.4.1 |

**よくある落とし穴**: AI 固有のイベントと広範な SIEM アラートとの関連付けを行なっていません。ドリフトを継続的な監視ではなく、スケジュールされたチェックとして扱っています。インシデント発生時に AI 固有のフォレンジックツールが不足しています。

## AD.19 人間による監督とシャットダウン制御

影響度の高いアクションには人間の承認を必須とし、人間の制御下で信頼性の高い、検証済みのシャットダウンおよび段階的な機能低下（グレイスフル デグラデーション）への移行経路を提供します。

| 管理策/手法 | 要件 ID |
| --- | --- |
| すべてのアクティブなエージェント インスタンスを停止するスウォーム レベルのキル スイッチを実装します。 | C9.1.3 |
| 明示的な人間の承認が得られ、検証されるまで、特権アクション、影響度の高いアクション、または不可逆的なアクションをランタイムでブロックします。 | C9.2.1 |
| 承認要求には、切り捨てなしで正規化された完全なアクション パラメータ（差分、コマンド、受信者、量、リソース、スコープ）を表示します。 | C9.2.2 |
| 各高影響アクションに対して、信頼できる可逆性分類（読み取り専用、可逆、外部可逆、不可逆）を実施します。 | C9.2.3 |
| 可逆性分類を実行時に適用します（ブロック、承認必須、制限）。| C9.2.4 |
| 自己変更機能（プロンプト書き換え、ツール リスト変更、パラメータ更新）を制限し境界を設定します。| C9.2.5 |
| 決定論的ポリシー ゲートに加えて（置き換えはしません）、計画された高リスク アクションに対する AI によるレビューを実施します。| C9.2.6 |
| プロンプト インジェクションによる迂回に対する AI によるレビュー機構を保護します。| C9.2.7 |
| パラメータ、要求者のアイデンティティ、実行コンテキスト、およびシングル ユースのノンスに暗号的にバインドされた承認を実施します。 | C9.2.8 |
| 承認発行の鍵マテリアルまたは資格情報をエージェント実行時から分離します。 | C9.2.9 |
|多段階チェーンまたはマルチ エージェント チェーンにおいて、チェーン内で最も影響の大きい可逆性分類を適用します。 | C9.2.10 |
| モデル推論と出力を即座に停止する手動キル スイッチを実装します。 | C9.6.1 |
| 人間による承認ゲートが定義時間内に満たされない場合、保留中のアクションをフェイル クローズでブロックします。 | C9.6.2 |
| エージェント ランタイムから隔離された帯域外チャネルを介して、キル スイッチ コマンドを配信します。 | C9.6.3 |
| ローカル MCP サーバーのインストール時に、明示的な同意ダイアログとキャンセル オプションを提供します。 | C10.4.7 |

**よくある落とし穴**: ランタイム ゲートに一度も接続されたことのない高リスク アクション ポリシーを文書化している。承認をアイデンティティやコンテキストにバインドせずにパラメータにバインドしている。承認者が応答しない場合にデフォルトでフェイル オープンになっている。侵害されたエージェントに対して帯域内キル スイッチが機能すると想定している。実行されないキル スイッチを実装している。

# C8 メモリ、エンベディング、およびベクトル データベースのセキュリティ

## 管理策目標

この章では、検索拡張生成 (RAG) を通じて AI システムの半永続的および永続的な「メモリ」として機能するエンベディングとベクトル ストアのセキュリティについて説明します。

---

## C8.1 メモリおよび RAG インデックスに対するアクセス制御

すべてのベクトル コレクションに対して、きめ細かなアクセス制御とクエリ時のスコープ強制を適用する必要があります。

| # | 説明 | レベル |
| :--: | --- | :---: |
| **8.1.1** | ベクトル識別子と名前空間がテナントごとに一意性を強制し、テナント間の衝突を防止していることを**検証します**。| 1 |
| **8.1.2** | 文書のメタデータ タグが最初の書き込み後に不変であることを**検証します**。| 2 |
| **8.1.3** | 取得操作がスコープ制約を遵守していることを**検証します**。 | 2 |

---

## C8.2 エンベディングの無害化および検証

コンテンツはベクトル化の前に事前スクリーニングされ、メモリへの書き込みは信頼できない入力として扱われ、安全でないペイロードの取り込みを防ぐ必要があります。

| # | 説明 | レベル |
| :--: | --- | :---: |
| **8.2.1** | 機密フィールドがエンベディング化前に検出され、マスク、トークン化、または破棄されていることを**検証します**。 | 1 |
| **8.2.2** | 通常のクラスタリング パターンから外れたベクトルが、本番インデックスに入る前にフラグ付けされ、隔離されていることを**検証します**。 | 2 |
| **8.2.3** | エージェント出力とツール出力が、明示的なソース検証なしに、信頼できるエージェント メモリに自動的に書き込まれていないことを**検証します**。 | 2 |
| **8.2.4** | 検索結果を操作するために作成されたコンテンツが、ベクトル化の前に検出され、拒否または隔離されていることを**検証します**。| 3 |
| **8.2.5** | メモリに書き込まれた新しいコンテンツが、既に保存されているコンテンツとの矛盾がないかチェックされ、矛盾が発生した場合はアラートがトリガーされていることを**検証します**。| 3 |

---

## C8.3 メモリの有効期限と失効

メモリおよび RAG インデックスの保持と失効は、明示的かつ強制的に実行可能である必要があります。

| # | 説明 | レベル |
| :--: | --- | :---: |
| **8.3.1** | 有効期限切れのベクトルが検索結果から除外されていることを**検証します**。| 2 |
| **8.3.2** | メモリをリセットできることを**検証します**。| 2 |
| **8.3.3** | 隔離されたコンテンツが保持されるものの、すべての検索結果から除外されていることを**検証します**。| 3 |

# C12 監視、ログ記録および異常検知

## 管理策目標

この章では、AI 固有の脅威を検出しトリアージできるようにするために、モデルやその他の AI コンポーネントが何を見て、何を実行し、何を返すかをリアルタイムかつフォレンジック的に可視化することについて説明します。

---

## C12.1 リクエストとレスポンスのログ記録

監査証跡を作成しインシデント対応を支援するために、AI のリクエストとレスポンスをログに記録する必要があります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **12.1.1** | AI とのインタラクションがセッション コンテキストと AI 固有のテレメトリとともにログに記録されていることを**検証します**。 | 1 |
| **12.1.2** | 安全性フィルタリングとポリシー決定が、コンテンツ モデレーション システムの監査、デバッグ、フォレンジック分析を支援するのに十分な詳細情報とともにログに記録されていることを**検証します**。 | 2 |
| **12.1.3** | AI 推論イベントのログ エントリが、少なくともモデル識別子、トークン使用（入力と出力）、プロバイダ名、および操作タイプを含む、構造化された相互運用可能なスキーマに従っていることを**検証します**。| 2 |
| **12.1.4** | RAG パイプライン取得イベントが、クエリ、取得された文書、および知識ソースを含めてログに記録されていることを**検証します**。| 2 |

---

## C12.2 検出とアラート発行

AI 固有の攻撃パターン（脱獄（ジェイルブレイク）、プロンプト インジェクション、モデル抽出、マルチ ターン軌道攻撃、LLM エンドポイントを介した隠蔽チャネル）を検出し、下流の検出および対応システムが対応できるように、セキュリティ イベントに AI 固有のコンテキストを追加する必要があります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **12.2.1** | システムが既知の脱獄（ジェイルブレイク）パターン、プロンプト インジェクションの試み、および悪意のある入力を検出してアラートを発行しているを**検証します**。| 1 |
| **12.2.2** | 振舞い異常検知が異常な会話パターン、過剰な再試行、またはプロービング動作を識別していることを**検証します**。| 2 |
| **12.2.3** | カスタム ルールが、協調的な脱獄ジェイルブレイク）の試み、プロンプト インジェクション、およびシステム プロンプト抽出の試みに関する AI 固有の脅威パターンを検出していることを**検証します**。| 2 |
| **12.2.4** | 調査を支援することを目的として、抽出のアラート イベントに問題のあるクエリ メタデータが含まれていることを**検証します**。| 2 |
| **12.2.5** | トークンの使用状況が、ユーザーごと、セッションごと、機能エンドポイントごと、チームまたはワーク スペースごとなど、詳細な属性レベルで追跡されていることを**検証します**。| 2 |
| **12.2.6** | マルウェアおよびコマンド＆コントロール（C2）活動を特定するために、LLM APIトラフィックが隠しチャネルの指標と通信シグネチャについて監視されていることを**検証します**。| 3 |

## C12.3 モデル、データ、およびパフォーマンス ドリフトの検出

モデル出力、入力分布、およびデータ スキーマ全体にわたるドリフトと劣化を監視し、品質低下とセキュリティに関連する動作変化を特定する必要があります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **12.3.1** | 入力データ タイプに合わせた統計的に検証された手法（例: 表形式の数値特徴量の場合は KS 検定または PSI、テキストまたは画像の場合はエンベディング距離指標）を使用して、データ ドリフト検出が、モデルのパフォーマンスに影響を与える可能性のある入力分布の変化を監視していることを**検証します**。| 1 |
| **12.3.2** | 幻覚（ハルシネーション）検出の監視が、事実と異なる、矛盾する、または捏造された情報を含むモデル出力を特定し、フラグを立てていることを**検証します**。| 2 |
| **12.3.3** | トレンド分析と持続的なモデル劣化の検出を可能にするために、幻覚（ハルシネーション）発生率が連続的な時系列指標として追跡されていることを**検証します**。| 2 |
| **12.3.4** | 説明のつかない動作の変化が徐々に発生する想定される運用上のドリフトと区別されていることを**検証します**。| 3 |

## C12.4 プロアクティブなセキュリティ動作監視

プロアクティブな（エージェントが開始する）動作から生じるセキュリティ脅威は、実行前の検証、動作パターン分析、およびセキュリティ上重要なアクションの承認のための監査証跡を含め、検出および防止される必要があります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **12.4.1** | 自律的なアクションのトリガーに、プロアクティブな行動パターン分析、セキュリティ評価、および脅威状況評価が含まれていることを**検証します**。| 2 |
| **12.4.2** | 監査ログに、承認者のアイデンティティ、タイムスタンプ、アクション パラメータ、および決定結果を含む、セキュリティ上重要なプロアクティブなアクションが記録されていることを**検証します**。| 2 |
| **12.4.3** | キル スイッチの有効化とオーバーライド コマンドがログに記録されていることを**検証します**。| 2 |

---

## C12.5 学習データとモデルのライフサイクル監査

学習データ、モデル アーティファクト、および知識ソースの出所と変更履歴は、AI 開発ライフサイクル全体を通して監査可能である必要があります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **12.5.1** | データセットの系統（リネージ）に、すべての変換、拡張、マージを含む、各データセットとそのコンポーネントが記録されていることを**検証します**。| 1 |
| **12.5.2** | すべてのラベル付けアクティビティがログに記録されていることを**検証します**。| 1 |
| **12.5.3** | すべてのモデル変更において変更不可能な監査レコードが生成されていることを**検証します**。| 2 |
| **12.5.4** | 書き込み時に、取り込まれたすべての文書に、ソース、作成者のアイデンティティ、タイムスタンプがタグ付けされていることを**検証します**。| 2 |

# C2 入力の確認（validation)

## 管理策目標

この章では、AI システムに対する最も深刻な攻撃の 1 つであるプロンプト インジェクション攻撃に対する第一線の防御策として、すべての入力の確認 (validation) について解説します。

---

## C2.1 プロンプト インジェクション対策

プロンプト インジェクションは、AI システムにとって最大の脅威の 1 つであり、これを防ぐにはパターン フィルタ、データ分類器、命令階層の強制といった複数の対策を組み合わせる必要があります。

| # | 説明 | レベル |
| :--------: | --------------------------------------------------------------------------------------------------------------------- | :---: |
| **2.1.1** | 入力の正規化がトークン化またはエンベディング化の前に適用されていることを**検証します**。 | 1 |
| **2.1.2** | 入力内のエンコードと表現の偽装（スマグリング）が検出され軽減されていることを**検証します**。承認された軽減策には、正規化、厳密なスキーマ検証、ポリシー ベースの拒否、または明示的なマーキングが含まれています。 | 1 |
| **2.1.3** | モデルの動作を制御する可能性のあるすべての入力が信頼できないものとして扱われ、プロンプト インジェクション検出ルールセットまたは分類子によって選別され、フラグ付きの入力がブロックされていることを**検証します**。 | 1 |
| **2.1.4** | 入力長の制御により、コンテンツがコンテキスト ウィンドウを超えることを防止していることを**検証します**。入力長の制御では、トークン制限を超える入力を切り捨てるのではなく、拒否する必要があります。 | 1 |
| **2.1.5** | システムがすべての入力に対して文字セット制限を実装していることを**検証します**。文字セット制限には、明示的に必要な文字のみを許可する許可リストのアプローチを使用する必要があります。 | 1 |
| **2.1.6** | ユーザーの指示が処理された後でも、システムと開発者のメッセージがユーザーの指示やその他の信頼できない入力をオーバーライドする指示階層がシステムによって強制されていることを**検証します**。 | 2 |
| **2.1.7** | 予約済みの特殊なトークンはリテラル文字としてエンコードされており、モデル コンテキストに挿入できないことを**検証します**。 | 2 |
| **2.1.8** | システムがマルチ ショット ジェイルブレイク（脱獄） パターンを検出できることを**検証します**。 | 3 |

---

## C2.2 コンテンツとポリシーのスクリーニング

構文的に有効なプロンプトであっても、ポリシー違反の指示、有害なコンテンツ、制限された情報など、許可されていないコンテンツを要求する場合があります。入力側のコンテンツ スクリーニングにより、このようなプロンプトがモデルに到達するのを防ぎます。

| # | 説明 | レベル |
| :--------: | --------------------------------------------------------------------------------------------------------------------- | :---: |
| **2.2.1** | すべてのプロンプトが、暴力、自傷行為、憎悪、性的コンテンツについて、設定可能なしきい値に基づいてコンテンツ分類器によってスコアリングされていることを**検証します**。これらのしきい値を超えるプロンプトは、モデル コンテキストに到達する前に拒否または無害化されています。 | 1 |
| **2.2.2** | サポートされていない言語についても、プロンプトのコンテンツ分類が評価されていることを**検証します**。 | 1 |
| **2.2.3** | 非テキスト入力（画像/動画/音声）について、敵対的摂動、ステガノグラフィ ペイロード、隠蔽または埋め込みコンテンツ、既知の攻撃パターンがないかを確認していることを**検証します**。| 2 |
| **2.2.4** | 複数の入力タイプにまたがる協調攻撃（例: 画像内のステガノグラフィ ペイロードとテキスト内のプロンプト インジェクションの組み合わせ）が検出され、ブロックされていることを**検証します**。| 3 |

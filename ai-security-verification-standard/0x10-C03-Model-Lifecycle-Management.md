# C3 モデルのライフサイクル管理および変更管理

## 管理策目標

この章では、不正な変更や安全でない変更が本番環境に到達しないようにすることを目的とした、モデルの変更に関する管理策について説明します。

---

## C3.1 モデルの承認と完全性

完全性が検証された承認済みモデルのみが、本番環境に到達できるようにする必要があります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **3.1.1** | モデル レジストリがデプロイされたすべてのモデル アーティファクトとその出所に関するインベントリを保持していることを**検証します**。| 1 |
| **3.1.2** | すべてのモデル アーティファクト（重み、構成、トークナイザー、ベース モデル、微調整（ファイン チューン）、アダプタ、および安全性/ポリシー モデル）が正規の機関によって暗号署名されていることを**検証します**。| 2 |
| **3.1.3** | モデルの暗号署名がデプロイの承認時とロード時に検証されていることを**検証します**。| 2 |

---

## C3.2 モデルの確認 (validation) とテスト

モデルは、デプロイ前に、既定のセキュリティおよび安全性の確認 (validation) に合格する必要があります。

| # | 説明 | レベル |
| :--------: | --------------------------------------------------------------------------------------------------------------- | :---: |
| **3.2.1** | モデルがデプロイ前に自動入力検証テスト、安全性評価テスト、および出力無害化テストを受けていることを**検証**します。| 1 |
| **3.2.2** | 学習後の量子化処理を受けたモデルが、デプロイ前に、圧縮されたアーティファクトに対して同じ安全性および整合性（アライメント）テスト スイートで再評価されていることを**検証**します。| 2 |
| **3.2.3** | プロバイダ モデル、バージョン、またはルーティングの変更が、継続使用前にセキュリティの再評価をトリガーしていることを**検証**します。| 3 |

---

## C3.3 制御されたデプロイメントとロールバック

ライフサイクル管理をサポートするために、モデルのデプロイメントは、制御および監視され、可逆性を備えている必要があります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **3.3.1** | 本番環境のデプロイメントにおいて、自動ロールバック トリガーを備えたロールアウト機構が実装されていることを**検証します**。 | 2 |
| **3.3.2** | ロールバック機能によって、モデルの状態が完全に復元されることを**検証します**。 | 2 |
| **3.3.3** | 並行して実行されるモデルのバージョンは、分離されたランタイム状態を使用し、AI 固有の共有リソースがデプロイメント間で共有されないようになっていることを**検証します**。 | 2 |

---

## C3.4 セキュアな開発慣行

モデル開発環境は、本番環境から分離する必要があります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **3.4.1** | AI 固有のランタイム コンポーネントが環境境界（開発、ステージング、本番など）を越えて共有されていないことを**検証します**。 | 1 |
| **3.4.2** | モデルの学習および微調整（ファイン チューニング）環境が本番環境から分離されていることを**検証します**。 | 2 |

---

## C3.5 微調整（ファイン チューニング）パイプライン

微調整（ファイン チューニング）パイプラインは、デプロイされたモデルの動作を大規模に変更する可能性のある、高い権限を必要とする操作です。多段階パイプラインでは、中間段階で何らかの侵害が発生すると、後続の段階が受け入れる微妙に改変されたアーティファクトが生成されるため、このリスクがさらに高まります。

| # | 説明 | レベル |
| :--------: | ------------------------------------------------------------------------------------------------------------------- | :---: |
| **3.5.1** | RLHF 微調整（ファイン チューニング）で使用されるモデルが、学習の実行前にバージョン管理され、完全性が照合されていることを**検証します**。| 2 |
| **3.5.2** | RLHF 学習段階に、報酬ハッキングまたは報酬モデルの過剰最適化を自動的に検出する機能が含まれていることを**検証します**。| 3 |
| **3.5.3** | 多段階の微調整（ファイン チューニング）パイプラインにおいて、各段階の出力が次の段階で消費される前に完全性照合が実施されていることを**検証します**。| 3 |
| **3.5.4** | 微調整（ファイン チューニング）チェックポイントが個別のアーティファクトとして登録されていることを**検証します**。| 3 |
